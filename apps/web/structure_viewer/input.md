# workspace_bootstrap
- 모노레포 루트는 워크스페이스 등록과 공통 설정의 기준점이어야 한다.
- 초기 범위는 structure_viewer 동작 검증에 필요한 최소 워크스페이스 구조로 제한한다.
- 특정 패키지 매니저 우회 동작에 과도하게 결합되지 않아야 한다.
> 모노레포 루트와 앱 및 공유 패키지 경계를 구성한다.
> Astro, React, TypeScript 기반 앱 시작 구조를 연결한다.

# monorepo_root
- 공통 설정은 Astro 앱과 지원 패키지에서 함께 재사용 가능해야 한다.
- 워크스페이스 등록 정보는 discovery 대상 범위를 명확히 표현해야 한다.
- 루트 도메인은 뷰 전용 상태를 직접 소유하지 않아야 한다.
> 워크스페이스 루트 설정을 초기화한다.
> 공통 TypeScript 및 빌드 규칙과 discovery용 메타데이터를 노출한다.

# domain_discovery
- 등록된 워크스페이스 도메인과 모듈 정보를 수집해 안정적인 타입 구조로 정규화해야 한다.
- 일부 메타데이터가 없어도 빈 컬렉션으로 수렴해야 한다.
- discovery 로직은 렌더링 계층과 분리된 순수 도메인 로직이어야 한다.
> 등록된 워크스페이스 범위를 스캔한다.
> 도메인, 모듈, 함수 리스트를 정규화해 상태 계층으로 전달한다.

# state_management
- Zustand는 발견된 도메인, 모듈, 함수 목록과 선택 및 필터 상태의 단일 클라이언트 저장소여야 한다.
- 파생 상태는 정규화된 discovery 데이터로부터 계산해야 한다.
- 동일한 discovery payload를 여러 위치에 중복 보관하지 않아야 한다.
> discovery 결과를 저장소에 적재한다.
> 선택 상태, 필터 상태, 패널 상태를 갱신하고 UI와 렌더러에 제공한다.

# module_card_ui
- 초기 모듈 카드 UI는 모듈 구조와 함수 목록의 가독성을 우선해야 한다.
- UI 컴포넌트는 React 기반으로 유지하고 정규화된 상태를 소비해야 한다.
- 빈 도메인, 빈 모듈, 로드 직후 상태도 깨지지 않고 렌더링되어야 한다.
> 선택된 모듈 요약과 카드 레이아웃을 렌더링한다.
> 상태 변화에 따라 빈 상태와 선택 상태를 반영한다.

# function_list_view
- 함수 리스트는 발견된 모듈 데이터로만 생성되어야 하며 하드코딩된 예시값을 사용하지 않아야 한다.
- 함수 없음 상황도 유효한 화면 상태로 처리해야 한다.
- 함수 리스트는 모듈 카드 내부에서 검증 가능한 최소 조회 화면이어야 한다.
> 선택된 모듈의 함수 식별자와 목록을 읽는다.
> 함수 리스트를 카드 내부에 렌더링하고 선택 변화에 동기화한다.

# rendering_runtime
- PixiJS는 구조 시각화의 2D 런타임으로만 사용해야 한다.
- React Three Fiber는 선택적 3D 시각화 런타임으로만 사용해야 한다.
- 렌더러가 비활성화되거나 실패해도 모듈 카드 UI는 계속 동작해야 한다.
> PixiJS와 React Three Fiber 연결 경계를 초기화한다.
> 정규화된 상태와 선택 상태를 구독해 시각 계층에 반영한다.
